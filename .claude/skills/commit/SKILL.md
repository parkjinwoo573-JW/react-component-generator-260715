# Commit

## 목적
변경사항을 분석해서 프로젝트 컨벤션에 맞는 커밋 메시지를 작성하고, 사용자 승인 후 커밋을 실행한다. 목표는 "그냥 커밋되게 하는 것"이 아니라, 나중에 `git log`만 봐도 무엇이 왜 바뀌었는지 알 수 있는 히스토리를 남기는 것이다.

## 절차

### 1. 변경사항 파악
다음을 병렬로 실행해서 현재 상태를 파악한다.
- `git status` — 추적되지 않은 파일을 포함한 전체 변경 파일 목록 (대형 저장소에서 메모리 문제를 일으킬 수 있는 `-uall` 플래그는 쓰지 않는다)
- `git diff` — 스테이징되지 않은 변경사항
- `git diff --staged` — 이미 스테이징된 변경사항
- `git log --oneline -10` — 이 저장소의 커밋 메시지 스타일(타입 prefix 유무, 한국어/영어, 길이 등)을 파악해서 그 스타일을 따른다

### 2. 논리적 단위로 분류
모든 변경사항을 하나의 커밋에 욱여넣지 않는다. 서로 관련 없는 변경(예: 기능 추가 + 포맷팅 전용 수정 + 문서 업데이트)이 섞여 있으면 커밋을 나눌지 판단한다.

- 하나의 목적으로 묶이는 변경끼리 하나의 커밋으로 만든다. 파일이 여러 개라도 목적이 하나면 한 커밋으로 충분하다.
- 나눌지 애매하면 사용자에게 물어본다. 임의로 여러 커밋을 만들어버리면 사용자가 의도한 것과 다른 히스토리가 생길 수 있다.

### 3. 커밋 메시지 작성
형식: `<type>: <한국어 요약>`

type은 변경의 성격에 따라 고른다.
- `feat`: 새 기능 추가
- `fix`: 버그 수정
- `refactor`: 동작 변화 없는 구조 개선
- `chore`: 빌드/설정/의존성 등 잡무성 변경
- 그 외 `docs`, `test`, `style` 등 저장소 관례에 맞는 타입이 있으면 따른다

요약은 diff만 봐도 알 수 있는 "무엇을 했는지"보다, 되도록 그 변경이 존재하는 이유(왜)가 드러나도록 쓴다. 한 줄로 충분하지 않으면 빈 줄 다음에 본문을 덧붙여 설명한다.

### 4. 사용자 승인
커밋을 실행하기 전에 반드시 다음을 제시하고 승인을 받는다.
- 작성한 커밋 메시지(들)
- 각 커밋에 포함될 파일 목록

여러 커밋으로 나누기로 했다면 각각을 구분해서 보여준다. 사용자가 메시지나 분류 수정을 요청하면 반영한 뒤 다시 확인받는다. 승인 없이는 5단계로 넘어가지 않는다.

### 5. 커밋 실행
승인을 받은 후에만 실행한다.
- 파일은 이름을 지정해서 스테이징한다. `git add -A`나 `git add .`는 의도치 않은 파일(비밀키, 빌드 산출물 등)을 포함시킬 위험이 있으므로 피한다.
- 커밋 메시지는 항상 heredoc으로 전달해서 줄바꿈과 특수문자가 깨지지 않게 한다.

```bash
git commit -m "$(cat <<'EOF'
<type>: <요약>

<필요시 본문>
EOF
)"
```

- 커밋 완료 후 `git status`로 정상적으로 반영됐는지 확인한다.

## 하지 말아야 할 것
- 사용자 승인 없이 커밋을 실행하지 않는다.
- `--no-verify`, `--no-gpg-sign` 등으로 훅이나 서명을 건너뛰지 않는다. pre-commit 훅이 실패하면 원인을 고치고 다시 커밋한다.
- 기존 커밋을 `--amend`하지 않는다 (사용자가 명시적으로 요청한 경우는 예외). 항상 새 커밋을 만든다.
- `.env`, 인증정보 파일처럼 비밀 정보가 담겼을 수 있는 파일은 파일명만 보고 넘어가지 말고 내용을 확인한 뒤, 의심스러우면 커밋 전에 사용자에게 알린다.
- 사용자가 명시적으로 요청하지 않는 한 push는 하지 않는다.
